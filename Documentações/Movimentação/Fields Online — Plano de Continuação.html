DM-01 Correção de Velocidade Diagonal (Fator √2) ● ALTA
⚠ Problema Movimento diagonal envia pacotes com mesmo stop_count que cardinal, fazendo personagem cobrir ~41% mais distância/segundo na diagonal.
✓ Solução Cliente adiciona delay extra com multiplicador 1/√2 ≈ 0.707. Servidor já foi ajustado com HD-01 (0.240s).
DM-02 Rate Limiting no Input Diagonal ● ALTA
⚠ Problema Input diagonal pode ser enviado múltiplas vezes no mesmo frame por conta do sistema de input do RGSS3.
✓ Solução Adicionar cooldown de input (last_move_frame) para garantir apenas 1 pacote por frame de movimento.
DM-03 Previsão Local de Passabilidade Diagonal ● ALTA
⚠ Problema Cliente envia qualquer diagonal ao servidor sem checar se é passável. Servidor rejeita, mas cliente já animou o passo → rubber-banding visual.
✓ Solução Chamar passable?(x, y, horz) && passable?(x, y, vert) antes de send_movement(diagonal). Só envia se ambas as direções forem passáveis.
DM-04 Sincronização de Direção ao Parar ● MÉDIA
⚠ Problema Ao soltar a tecla diagonal, o sprite pode ficar com facing incorreto (diagonal não é direção válida para facing no VXA-OS).
✓ Solução Ao parar, converter direção diagonal para a direção cardinal dominante (última tecla pressionada).
DM-05 Diagonal Bloqueada por Paredes em L ● BAIXA
⚠ Problema Em cantos de parede em formato L, o jogador consegue "escorregar" através do canto se ambas as direções individualmente forem passáveis.
✓ Solução Verificar também a tile diagonal diretamente com passable?(x+dx, y+dy, 0) como check adicional.

SM-01 Lerp de NetPlayers (outros jogadores) ● CRÍTICA
⚠ Problema Outros jogadores online ainda se "teletransportam" tile a tile sem interpolação visual, pois o lerp só está aplicado ao jogador local.
✓ Solução Aplicar smooth_x/smooth_y nos sprites de Game_NetPlayer usando o mesmo sistema de lerp já implementado.
SM-02 Snap em Teleportes (Anti-Warp Visual) ● ALTA
⚠ Problema Quando Game_NetPlayer sofre teleporte, o lerp interpola através do mapa inteiro causando "deslize" visual caótico.
✓ Solução Detectar distância > 3 tiles entre posição visual e lógica e aplicar snap direto (smooth_x = real_x imediatamente).
SM-03 Lerp Speed Dinâmica por Velocidade do Personagem ● MÉDIA
⚠ Problema Personagem rápido (move_speed alto) tem lerp lento demais — posição visual fica sempre "atrás" da lógica, parecendo lag.
✓ Solução Calcular lerp_speed como base_speed * (move_speed / 4.0) para transições proporcionais.
SM-04 Detecção Real de MOBs (via Nota do Evento) ● MÉDIA
⚠ Problema MOB_LERP_SPEED existe na config mas nunca é usado — todos os eventos usam EVENT_LERP_SPEED, incluindo MOBs.
✓ Solução Verificar @event.note ou nome do evento por tag <mob> para aplicar MOB_LERP_SPEED automaticamente.
SM-05 Inicialização Lazy e Segura de smooth_x/y ● BAIXA
⚠ Problema Em edge cases de teleporte/carregamento, @smooth_x pode ser nil se init_public_members não foi chamado antes de update.
✓ Solução Adicionar guard @smooth_x ||= @x.to_f no início de fo_update_smooth_position.
