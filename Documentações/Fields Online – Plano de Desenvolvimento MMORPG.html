🎯 Público-Alvo

Fields Online é um MMORPG 2D top-down desenvolvido com RPG Maker VX Ace e o framework VXA-OS, que transforma o engine em uma plataforma multiplayer robusta.

O jogo combina a simplicidade visual dos RPGs clássicos 2D com mecânicas profundas de MMORPGs modernos, criando uma experiência nostálgica porém rica em conteúdo.

Inspirado em Tibia, Ragnarok Online, MU Online, Diablo e World of Warcraft, Fields Online busca capturar a essência de cada um desses títulos em um mundo coeso e original.

🎮 Faixa etária: 16–35 anos
🕹️ Perfil: Jogadores nostálgicos de MMORPGs clássicos
🌐 Idioma: Português Brasileiro
💻 Plataforma: PC Windows (offline/LAN/online)

⚠️ Limitações
  • ⚠Engine single-threaded limita processamento paralelo
  • ⚠RGSS3 baseado em Ruby 1.9.x (versão antiga)
  • ⚠Resolução fixa de 544×416 pixels por padrão
  • ⚠Sem suporte nativo a shaders ou efeitos 3D
  • ⚠Limite de tiles e spritesheets por mapa
  • ⚠Performance degrada com muitos eventos simultâneos
  • ⚠Sem suporte nativo a IPv6
  • ⚠Documentação do VXA-OS limitada e desatualizada
✨ Potencialidades
  • ✦Altamente customizável via scripts Ruby
  • ✦Comunidade ativa com vasto acervo de scripts
  • ✦VXA-OS já implementa base sólida de MMO
  • ✦Desenvolvimento rápido com editor visual integrado
  • ✦Baixo custo de hardware para servidor
  • ✦Estética 2D pixel art com apelo nostálgico
  • ✦Fácil criação de conteúdo (mapas, eventos, NPCs)
  • ✦Possibilidade de expansão modular do servidor

Sistema de Combate

Tibia Diablo MU Online Ragnarok

📋 Mecânicas Principais

Combate em Tempo Real: Ataques automáticos com cooldown configurável por arma. Habilidades ativas com custo de MP e cooldown individual.
Fórmula de Dano: DMG = (ATK × 1.5) - (DEF × 0.8) + Random(±15%). Crítico: 2× dano com chance baseada em AGI.
Status Effects: Veneno, Paralisia, Silêncio, Cegueira, Maldição, Queimadura, Congelamento, Atordoamento.
Aggro System: Monstros perseguem o jogador com maior ameaça. Tanks podem usar habilidades de provocação.

💻 Código Ruby — Sistema de Cooldown

Ruby/RGSS3 #============================================================================== # [FO] Skill Cooldown System # Fields Online — Sistema de Cooldown de Habilidades #============================================================================== module FO module Cooldown # Cooldowns em frames (60 frames = 1 segundo) SKILL_COOLDOWNS = { 1 => 120, # Fireball: 2s 2 => 180, # Ice Bolt: 3s 3 => 300, # Meteor: 5s 10 => 60, # Slash: 1s 11 => 240, # Whirlwind: 4s } end end class Game_Actor attr_reader :skill_cooldowns alias fo_cooldown_init initialize def initialize(actor_id) fo_cooldown_init(actor_id) @skill_cooldowns = {} end def skill_on_cooldown?(skill_id) (@skill_cooldowns[skill_id] || 0) > 0 end def use_skill_cooldown(skill_id) cd = FO::Cooldown::SKILL_COOLDOWNS[skill_id] || 0 @skill_cooldowns[skill_id] = cd end def update_cooldowns @skill_cooldowns.each do |id, frames| @skill_cooldowns[id] = [frames - 1, 0].max end end def cooldown_percent(skill_id) max = FO::Cooldown::SKILL_COOLDOWNS[skill_id] || 1 current = @skill_cooldowns[skill_id] || 0 (current.to_f / max * 100).to_i end end

⚠️ Seção 7 — Considerações e Riscos

Análise de riscos do projeto e boas práticas de desenvolvimento.

🎲 Tabela de Riscos
RiscoProbabilidadeImpactoMitigação
Performance degradada com muitos jogadores 🔴 Alta Alto Testes de stress contínuos, otimização de queries
Bugs de sincronização cliente-servidor 🔴 Alta Alto Testes unitários, validação server-side de todas as ações
Limitações do RGSS3 bloqueando features 🟡 Média Médio Pesquisa prévia de viabilidade, workarounds criativos
Escopo crescente (feature creep) 🔴 Alta Médio Backlog priorizado, foco nas fases definidas
Conflitos entre scripts [RM], [VS] e [FO] 🟡 Média Médio Testes de integração, uso de alias em vez de override
Perda de dados do banco MySQL 🟢 Baixa Alto Backups automáticos diários, replicação
Cheating e exploits de jogadores 🟡 Média Médio Validação server-side, sistema anti-cheat básico
Desbalanceamento de classes/economia 🔴 Alta Médio Testes extensivos, feedback de beta testers
Atraso no cronograma por complexidade 🟡 Média Médio Buffer de 20% no cronograma, priorização de MVP
Incompatibilidade com versões futuras do VXA-OS 🟢 Baixa Médio Versão fixada no repositório, testes antes de atualizar
📌 Boas Práticas de Desenvolvimento
  1. Sempre validar ações no servidor — nunca confiar no cliente para lógica crítica
  2. Usar alias em vez de sobrescrever métodos do VXA-OS para evitar conflitos
  3. Comentar todos os scripts com cabeçalho padronizado [FO] e descrição de dependências
  4. Fazer commits frequentes com mensagens descritivas seguindo Conventional Commits
  5. Testar cada sistema isoladamente antes de integrar com outros módulos
  6. Manter um changelog atualizado a cada versão do projeto
  7. Usar constantes configuráveis no início de cada script para facilitar ajustes
  8. Implementar logging no servidor para rastrear erros e comportamentos anômalos
  9. Realizar backups do banco de dados antes de qualquer migração de schema
  10. Documentar todos os opcodes de rede com direção, formato e exemplos
  11. Seguir abordagem incremental: simples primeiro, complexidade depois
  12. Criar scripts modulares e reutilizáveis seguindo o template [FO]
  13. Testar com múltiplos clientes simultâneos desde as primeiras fases
  14. Manter separação clara entre lógica de cliente (UI) e lógica de servidor (gameplay)
