🎨 [FO] Smooth_Movement — Cliente
Interpolação de posição visual · Lerp de personagens, NPCs e MOBs
5 melhorias identificadas
#SM-01 ALTA Lerp Frame-Rate Independent
⚠ Problema
A interpolação dx * speed é dependente do framerate. A 30fps o lerp é mais lento que a 60fps — o personagem "arrasta" diferente dependendo da máquina.
✓ Solução
Multiplicar o fator de lerp por Graphics.delta_time (tempo real por frame), tornando a suavização idêntica em qualquer framerate.
[FO] Smooth_Movement
#SM-02 ALTA Snap em Teleportes (Anti-Warp Visual)
⚠ Problema
Quando um Game_NetPlayer sofre teleporte (mudança brusca de mapa/posição), o lerp interpola através do mapa inteiro causando um "deslize" visual caótico.
✓ Solução
Detectar distância > 3 tiles entre posição visual e lógica e aplicar snap direto (smooth_x = real_x imediatamente), ignorando o lerp neste frame.
[FO] Smooth_Movement
#SM-03 MÉDIA Lerp Speed Dinâmica por Velocidade do Personagem
⚠ Problema
Um personagem rápido (move_speed alto) tem lerp lento demais — a posição visual fica sempre "atrás" da posição lógica, parecendo lag.
✓ Solução
Calcular lerp_speed como base_speed * (move_speed / 4.0) para que personagens mais rápidos tenham transições mais ágeis proporcionalmente.
[FO] Smooth_Movement
#SM-04 MÉDIA Detecção Real de MOBs (via Nota do Evento)
⚠ Problema
MOB_LERP_SPEED existe na config mas nunca é usado — todos os eventos usam EVENT_LERP_SPEED, incluindo MOBs, que deveriam ter lerp levemente diferente.
✓ Solução
Verificar @event.note ou o nome do evento por tag como <mob> para aplicar MOB_LERP_SPEED automaticamente.
[FO] Smooth_Movement
#SM-05 BAIXA Inicialização Lazy e Segura de smooth_x/y
⚠ Problema
Em edge cases de carregamento de save/teleporte, @smooth_x pode ser nil se init_public_members não tiver sido chamado antes de update.
✓ Solução
Adicionar guard @smooth_x ||= @x.to_f no início de fo_update_smooth_position para garantir inicialização segura em qualquer ordem.
[FO] Smooth_Movement
⌨️ [FO] Diagonal_Movement — Cliente
Input diagonal · Passabilidade · Rate limiting · Sincronização
5 melhorias identificadas
#DM-01 ALTA Correção de Velocidade Diagonal (Fator √2)
⚠ Problema
Mover-se na diagonal envia pacotes com o mesmo stop_count que o movimento cardinal, fazendo o personagem cobrir ~41% mais distância por segundo na diagonal (violação da física básica).
✓ Solução
Cliente: Adicionar delay extra de 1 frame ao enviar movimento diagonal com multiplicador 1/√2 ≈ 0.707. Servidor: Ajustar stop_count para 0.240s no diagonal vs 0.170s no cardinal.
[FO] Diagonal_Movement + handle_data.rb
#DM-02 ALTA Input Buffer de 2 Frames (Diagonal Tolerante)
⚠ Problema
Pressionar W e D com 1-2 frames de diferença não reconhece como diagonal — o personagem move-se para cima por 1 frame, depois vira diagonal. Parece "escorregamento".
✓ Solução
Manter um buffer de estado das teclas dos últimos 2 frames. Se W e D foram pressionados dentro de 2 frames, tratar como diagonal imediatamente.
[FO] Diagonal_Movement
#DM-03 ALTA Previsão Local de Passabilidade Diagonal
⚠ Problema
O cliente envia qualquer diagonal ao servidor sem checar se é passável. O servidor rejeita, mas o cliente já animou o passo localmente → rubber-banding visual.
✓ Solução
Chamar passable?(x, y, horz) && passable?(x, y, vert) antes de send_movement(diagonal). Só envia ao servidor se ambas as direções forem passáveis.
[FO] Diagonal_Movement
#DM-04 MÉDIA Rate Limiting de Input no Cliente
⚠ Problema
Não há nenhum controle de frequência de envio de pacotes no cliente. Em lag ou loop rápido, pode inundar o servidor com pacotes PLAYER_MOVE.
✓ Solução
Adicionar @move_send_cooldown = 0 e só enviar send_movement se Time.now >= @move_send_cooldown, então definir @move_send_cooldown = Time.now + 0.15.
[FO] Diagonal_Movement
#DM-05 BAIXA Guards Sincronizadas com o VXA-OS
⚠ Problema
As guards de move_by_input replicam manualmente as do [VS] Game_Player original. Se o VXA-OS atualizar, o script ficará desatualizado silenciosamente.
✓ Solução
Chamar fo_diag_original_move_by_input como fallback e interceptar apenas após a decisão de movimento, reduzindo a superfície de dependência.
[FO] Diagonal_Movement
🖥️ handle_data.rb — Servidor
Validação de pacotes · Antispam · Sincronização de posição
4 melhorias identificadas
#HD-01 ALTA stop_count Diferenciado para Diagonal
⚠ Problema
O servidor usa stop_count = Time.now + 0.170 para TODOS os movimentos. Na diagonal, o personagem percorre distância ~41% maior no mesmo tempo — é literalmente mais rápido em diagonal.
✓ Solução
stop_count = Time.now + (d.odd? ? 0.240 : 0.170) — diagonal espera mais antes do próximo movimento, normalizando a velocidade efetiva.
handle_data.rb
#HD-02 ALTA Rejeitar Direção 5 Explicitamente
⚠ Problema
A validação atual aceita d=5 (centro do numpad, não é direção), que passaria para move_straight(5) gerando comportamento indefinido no servidor.
✓ Solução
Adicionar return if d == 5 junto à validação de range, ou usar: return unless [1,2,3,4,6,7,8,9].include?(d) para whitelist explícita.
handle_data.rb
#HD-03 MÉDIA Antispam no Movimento
⚠ Problema
Não há verificação de antispam_time no movimento. Um cliente modificado pode enviar centenas de pacotes PLAYER_MOVE por segundo, sobrecarregando o servidor.
✓ Solução
Verificar client.antispam_time no início de handle_player_movement e atualizar após processamento (tempo mínimo entre pacotes: ~0.10s).
handle_data.rb
#HD-04 MÉDIA Confirmação de Posição Anti-Desync
⚠ Problema
Após movimento, o servidor não envia confirmação de posição ao cliente. Com pacotes perdidos ou movimentos rejeitados, a posição cliente/servidor diverge gradualmente (desync silencioso).
✓ Solução
Quando move_succeed for false, enviar send_player_position(client) para forçar correção no cliente e eliminar o desync acumulado.
